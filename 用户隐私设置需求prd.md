# 用户中心隐私设置需求 PRD

## 一、需求概述

### 1.1 需求背景
- 当前用户中心仅支持用户查看自己的个人信息，功能单一
- 缺乏社交属性，无法查看其他用户主页
- 需要增加更丰富的内容展示模块

### 1.2 需求目标
- **社交化**：支持用户访问其他用户主页，建立关注关系
- **隐私保护**：提供细粒度的隐私设置，用户可自主控制信息可见性
- **内容丰富**：增加多个内容展示模块（动态、交易、收藏、宠物图鉴等）
- **合规性**：符合GDPR/PIPL等隐私法规要求，确保用户知情同意

## 二、用户体系背景说明

本节说明网站用户体系的整体设计，作为后续功能的基础背景信息。

> **说明**：支持运营管理员在管理后台对网站用户进行管理操作（如授予VIP、添加标签等）。

### 2.1 用户角色

网站用户分为两种角色，决定用户的基础权限：

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| **VIP用户** | 特权用户（团队成员、KOL、游戏开发者等） | 享有平台特权功能，如创建抽奖、发布动态等 |
| **普通用户** | 默认角色 | 基础功能使用权限 |

- **新用户注册**：自动分配为"普通用户"角色
- **VIP授予**：由运营在管理后台搜索用户ID/昵称后手动授予

### 2.2 用户标签

用户标签用于**展示身份**，不控制权限。某些标签需由运营在管理后台为指定用户添加，或设计机制化的标签（来源于数据条件），可持续丰富

| 标签 | 英文名称 | 颜色 | 说明 |
|------|----------|------|------|
| 平台官方 | **Official** | 紫色 | 平台创始团队成员 |
| 内容创作者 | **Influencer** | 金色 | KOL、社区活跃内容创作者 |
| 游戏开发者 | **Game Developer** | 绿色  | Roblox小游戏开发者 |
| 资深用户 | **Veteran** | 蓝色 | 长期活跃的老用户（机制化标签） |
| 信誉交易者 | **Trusted Trader** | 桔色 | 交易信誉良好的用户（机制化标签） |

**展示规则**：
- **个人主页**：最多显示8个标签（超出不展示，按获得时间倒序）
- **其他场景**：显示3个标签 + "more"，点击more则跳转用户主页

### 2.3 功能权限

细粒度的功能权限控制，与用户角色配合使用。目前有2种功能权限：
- 创建抽奖活动（本次开发）
- 发布动态post（后续迭代）

| 权限 | 说明 | VIP默认 | 普通用户默认 |
|------|------|---------|--------------|
| **create_giveaway** | 创建抽奖活动 | ✅ 有 | ❌ 无 |
| **create_post** | 发布动态post | ✅ 有 | ❌ 无 |

**权限优先级**：`手动设置 > 角色默认`
- 普通用户可通过申请获得单独权限（如通过Discord频道申请，入口在用户主页的动态中-空状态的默认按钮）
- 运营可在管理后台为任意用户手动开启/关闭特定权限

## 三、用户主页的不同状态
具体url路径由研发设计，以下涉及路径时均为示意。

### 3.1 主人态
- **访问权限**：仅用户本人
- **主要操作**：查看完整个人信息、编辑资料、设置隐私权限、查看私密内容
- **主按钮**：「设置」按钮（右上角）

### 3.2 访客态
- **访问权限**：所有用户（根据隐私设置控制可见内容）
- **主要操作**：查看公开信息、关注/取消关注用户
- **主按钮**：「关注」/「已关注」按钮（右上角）

### 3.3 自己的访客态
- **表现形式**：与主人态一致，显示「设置」按钮
- **说明**：如当用户访问 `/user/自己的ID` 时，系统自动重定向到 `/profile`（主人态），从而避免出现关注自己的场景。

## 六、关注功能详细设计

### 6.1 关注按钮

#### 6.1.1 按钮位置
- 在访客态时显示页面右上角

#### 6.1.2 按钮状态
**未关注状态**
- 图标+文案：「Follow」

**已关注状态**
- 图标+文案：「Followed」
- Hover时显示：「Unfollow」取消关注

**加载状态**（如耗时少则不需要）
- 文案：「Loading...」
- 按钮禁用，显示loading spinner


#### 6.1.3 关注/取消关注交互
1. 点击「关注」按钮
2. 按钮进入加载状态
3. 发送关注请求到后端
4. 成功后：
   - 按钮状态变为「Followed」
   - 用户的粉丝数+1
   - 当前访客的关注数+1
   - Toast提示："You are now following @username" （使用已有的提示组件）
5. 失败时：
   - 显示错误Toast，"Something went wrong. Please try again."
   - 按钮恢复到「关注」状态


#### 6.1.4 取消关注确认
- Hover「Followed」按钮时，文案变为「Unfollow」，背景变色
- 点击按钮后执行取消关注操作
- Toast提示：Unfollowed @username

## 八、首次更新通知与合规设计

### 8.1 通知触发时机
- 用户在本次功能上线后首次登录时触发
- 使用 `localStorage` 或数据库字段记录用户是否已查看通知
- 显示次数：最多2次
  - 第1次：首次登录时弹出
  - 第2次：第2次登录时弹出（如果第1次点击了"稍后再说"）
  - 如果用户两次都点击"稍后再说"，则不再主动弹窗

### 8.2 通知弹窗设计

#### 8.2.3 按钮说明

**前往设置**
- 样式：主要按钮（Go to Settings）
- 行为：
  - 关闭弹窗
  - 跳转到 `/settings/privacy` 页面
  - 记录用户已查看通知（不再重复显示）
  - 如为新用户，点击该按钮后拉起登录流程，完成登录后再跳转到 `/settings/privacy` 页面。如未完成登录，则回到跳转前的页面，弹窗消失。

**稍后再说**
- 样式：次要按钮（Maybe Later）
- 行为：
  - 关闭弹窗
  - 增加查看次数计数
  - 如果查看次数 < 2，下次登录时再次显示
  - 如果查看次数 >= 2，不再显示


### 8.3 数据存储方案

#### 8.3.1 方案：LocalStorage（推荐用于本次实现）
**优点**：
- 实现简单，无需数据库迁移
- 性能好，不增加服务器负担
**缺点**：（可接受）
- 清除浏览器数据会重置
- 无法跨设备同步


### 8.4 隐私合规说明

#### 8.4.1 合规原则
- **知情同意**：通过弹窗明确告知用户新功能和隐私设置
- **数据最小化**：敏感信息（如交易记录）默认仅自己可见
- **用户控制**：提供细粒度的隐私设置，用户可自主选择
- **透明度**：在设置页面提供清晰的说明文案
- **数据访问权**：用户可随时查看和修改自己的隐私设置

### 9.1 路由列表
| 路由 | 说明 | 访问权限 |
|------|------|---------|
| （研发确认）| 个人中心（主人态） | 仅登录用户本人 |
| （研发确认）| 用户主页（访客态/主人态） | 根据隐私设置控制 |
| `/settings/privacy` | 隐私设置页面 | 仅登录用户本人 |


### 9.2 跳转逻辑

#### 9.2.1 
- 未登录用户访问：可访问其他用户主页并查看公开的内容，但点击关注按钮时则重定向到登录页面，登录完成才可完成关注动作
- 已登录用户访问：展示和功能无额外限制

#### 9.2.2 
- 检查 `userId` 是否为当前登录用户
  - 是：展示主人态
  - 否：展示访客态

#### 9.2.3 隐私权限检查流程
```
访问 /user/:userId
    ↓
是否为本人？
    ↓ 是
  展示主人态
    ↓ 否
查询目标用户的隐私设置
    ↓
profile_visibility = ?
    ↓ public
  展示访客态（所有公开内容）
```
