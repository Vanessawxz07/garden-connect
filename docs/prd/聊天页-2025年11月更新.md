## 🔄 2026年3月更新：移动端交易状态内联改造

> **更新日期：** 2026-03-12
> **关联文档：** 交易链路优化-202603.md（成交概率提示）、信任体系与积分闭环-202603.md（评价触发、积分引导）

### 1. 改造背景与价值

#### 1.1 核心问题

移动端聊天界面中，**交易状态变更面板**（Accept/Decline、Start Trade、Complete/Having an Issue 等操作按钮组）以固定区域形式堆叠在订单信息区下方，随着交易推进状态越多，固定区域越高。实测数据：

| 固定区域 | 占用高度（估算） |
|---------|--------------|
| 导航栏 + 头部信息栏 | ~100px |
| 订单信息区（紧凑版） | ~80px |
| **交易状态操作面板**（当前方案） | **~150–250px**（随状态阶段变化） |
| 输入区域 | ~56px |
| **合计固定区域** | **~386–486px** |

在 667px 高度的 iPhone SE 屏幕上，**可见聊天区域仅剩 ~180px（约 3 行消息）**，严重影响沟通效率。

#### 1.2 改造价值

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 移动端可见聊天区域 | ~180px（~27%） | ~430px（~65%） |
| 交易操作可达性 | 始终可见但挤占空间 | 随对话流滚动，最新卡片自动定位到可视区 |
| 状态感知 | 固定面板，信息密度低 | 时间线式呈现，可回溯完整交易历程 |
| 跨功能扩展性 | 新增状态需扩大固定区 | 新系统消息类型直接插入对话流，零布局成本 |

#### 1.3 必要性分析

- **用户痛点强烈**：交易聊天的核心场景是沟通协商，固定面板过多直接降低沟通效率，用户需频繁滚动才能看到对方消息
- **竞品验证**：闲鱼、eBay Messages 等成熟交易平台均采用"状态变更内联对话流"的设计模式
- **架构收益**：内联卡片模式天然兼容信任体系PRD中的"评价引导卡片"和"积分提示卡片"，无需另建UI容器
- **桌面端无影响**：桌面端屏幕充裕，订单信息区保持现有固定展示不变，本次改造仅针对移动端

### 2. 改造方案

#### 2.1 移动端布局调整

**改造后固定区域：**

```
┌─────────────────────────┐
│ ← 用户名  Online  [Report] │  ← 头部信息栏（精简，~48px）
├─────────────────────────┤
│ 🖼🖼🖼 Trade #1234  View >│  ← 紧凑订单信息条（~48px）
├─────────────────────────┤
│                         │
│   （对话流 + 内联状态卡片）  │  ← 可滚动区域（~515px @iPhone SE）
│                         │
├─────────────────────────┤
│ [Message...        ] [➤]│  ← 输入区域（~56px）
└─────────────────────────┘
```

**固定区域缩减：** ~152px（头部48 + 订单条48 + 输入56），相比改造前减少 ~234–334px。

#### 2.2 紧凑订单信息条（移动端专用）

替代原有的大面积订单信息区，采用单行横向布局：

```
[🖼][🖼][🖼] Trade #1234          View Details >
 ↑ 最多展示3个道具缩略图（24x24px）
```

- 道具缩略图横向排列，超过3个显示 `+N`
- 点击 `View Details` 跳转订单详情页
- 桌面端保持原有的 4x3 网格展示不变

#### 2.3 交易状态内联系统消息卡片

将原固定面板中的交易状态操作，改为在对话流中以 **系统消息卡片** 形式呈现。每当订单状态发生变更，系统自动在对话流中插入对应卡片。

##### 2.3.1 系统消息卡片类型定义

| 卡片类型 (type) | 触发时机 | 包含操作按钮 | 说明 |
|----------------|---------|-------------|------|
| `trade_request` | 发起交易请求时 | ACCEPT / DECLINE | 对方收到交易请求 |
| `trade_accepted` | 接受交易后 | — | 纯状态通知 |
| `trade_declined` | 拒绝交易后 | — | 纯状态通知 |
| `trade_started` | 任一方点击 Start Trade | COMPLETE / HAVING AN ISSUE | 交易进行中 |
| `trading_guide` | 伴随 trade_started 自动插入 | — | 游戏内交易操作指引 |
| `trade_complete_request` | 一方点击 Complete | — | 通知对方确认完成 |
| `trade_completed` | 双方均确认完成 | — | 交易完成通知 |
| `trade_issue` | 点击 Having an Issue | — | 争议标记通知 |
| `scam_warning` | 进入聊天时自动插入（一次性） | ✕ 关闭 | 反欺诈安全提醒 |
| `review_prompt` | 订单完成后 | ⭐ Leave a Review | 评价引导（对接信任体系PRD §3.1） |
| `points_earned` | 确认完成/提交评价后 | — | 积分获得通知（对接信任体系PRD §4.2.3） |

##### 2.3.2 各阶段卡片展示规格

```

│      绿色          红色             │

```

**② Trade Started 卡片（type: `trade_started`）**

```
┌─ 🔔 ──────────────────────────────┐
│  🎮 Trade In Progress              │
│                                    │
│  Trade has started! Meet in-game   │

**③ Review Prompt 卡片（type: `review_prompt`）**

```

**④ Scam Warning 卡片（type: `scam_warning`）**

```

│                                    │

```

- 位于对话流最顶部（首条消息之前）
- 关闭后本次会话不再显示（localStorage 记录已关闭状态）

##### 2.3.3 按钮状态管理规则

**核心原则：** 卡片中的操作按钮必须与当前订单状态实时同步，防止过期操作。

| 场景 | 按钮行为 |
|------|---------|
| 订单已进入下一状态 | 原卡片按钮置灰 + 显示已处理文案（如 `Accepted ✓`） |
| 订单被取消/关闭 | 所有未处理按钮置灰 + 显示 `Trade Cancelled` |
| 当前用户已操作，等待对方 | 按钮替换为等待提示（如 `Waiting for @username to confirm...`） |
| 操作超时 | 按钮置灰 + 显示 `Expired` |

**实现要点：**
- 每张卡片通过 `trade_id + card_type` 关联订单当前状态
- 卡片渲染时从订单状态 store 读取最新状态，动态决定按钮样式
- 按钮点击事件通过统一的交易状态机处理，不在卡片组件内直接修改订单

#### 2.4 响应式策略

| 终端 | 订单信息区 | 交易状态操作 | 安全警告 |
|------|----------|------------|---------|
| **桌面端** | 保持现有固定展示（4x3 道具网格） | 保持现有固定面板 | 保持现有固定警告条 |
| **移动端** | 紧凑订单信息条（缩略图+标题） | **内联系统消息卡片**（本次改造） | **内联对话流顶部卡片**（可关闭） |

> **桌面端完全不受影响**，所有改造仅作用于移动端视图（断点 < 768px）。

### 3. 跨PRD对接说明

| 关联PRD | 对接点 | 对接方式 |
|---------|--------|---------|
| 信任体系与积分闭环 §3.1 | 交易完成后触发评价 | 在对话流插入 `review_prompt` 卡片，点击按钮打开评价弹窗 |
| 信任体系与积分闭环 §4.2.3 | 确认/评价后发放积分 | 在对话流插入 `points_earned` 卡片展示获得积分 |
| 信任体系与积分闭环 §5.2 | 聊天引导横幅 | 复用 `points_earned` 或独立引导卡片，替代原固定横幅 |
| 交易链路优化 §5 | 成交概率提示弹窗 | 弹窗在创建/报价时触发，与聊天页无直接关联，但聊天中的 trade_request 卡片可展示 ratio 参考信息（P2） |

### 4. 工作量评估

| 工作项 | 预估工时 | 优先级 | 说明 |
|--------|---------|--------|------|
| 紧凑订单信息条组件（移动端） | 0.5d | P0 | 纯UI组件，缩略图+标题横向布局 |
| 系统消息卡片基础框架 | 1d | P0 | 定义卡片类型接口、渲染容器、样式规范 |
| trade_request / trade_accepted / trade_declined 卡片 | 1d | P0 | 含按钮交互和状态同步逻辑 |
| trade_started / complete / issue 卡片 | 1d | P0 | 含按钮状态管理 |
| scam_warning 内联卡片 | 0.5d | P0 | 关闭状态持久化 |
| 按钮状态同步逻辑（订单状态 → 卡片按钮） | 1d | P0 | 核心难点，防止过期操作 |
| review_prompt / points_earned 卡片 | 0.5d | P1 | 依赖信任体系后端完成后对接 |
| 桌面端兼容性验证 | 0.5d | P0 | 确保桌面端无回归 |
| **合计** | **~6d** | | 可独立测试和上线，不依赖信任体系后端 |

**独立上线策略：**
- P0 部分（~5.5d）可独立完成和上线，仅涉及前端展示层改造
- P1 部分（review_prompt / points_earned）在信任体系后端就绪后再对接，通过卡片 type 扩展即可，零改动成本
- 桌面端代码不变，通过响应式断点隔离，风险极低

### 5. 后续迭代方向（P2）

- 卡片支持折叠/收起历史已处理卡片，减少对话流噪音
- trade_request 卡片展示交易 ratio 参考（对接交易链路优化PRD）
- 支持图片消息后，系统卡片可附带道具截图预览
- 长对话场景下的"跳转到最新状态卡片"快捷按钮

## 🔄 2026年3月更新：会话列表分游戏筛选改为下拉选择器

> **更新日期：** 2026-03-09
> **关联文档：** 新游拓展专项-2025年1月.md § 聊天页
> **改造原因：** 随着新游上线，横向Tab的空间无法容纳更多游戏筛选项，改为统一下拉选择器（一次性改造）。

### 1. 原方案（已废弃）

会话列表顶部使用横向 Tab 进行分游戏筛选：`All | GAG | BF | ...`

### 2. 新方案：统一下拉选择器

**触发器样式：**
```
[ ▼ All Message 🔴3 ]
```

**展开下拉菜单：**
```
[ ▼ All Message 🔴3 ]
   ─────────────────
 ✓ All Message        🔴3
   🌱 Grow a Garden   🔴1
   🍎 Blox Fruits     🔴2
   🌊 Escape Tsunami    
```

**交互规格：**

| 要素 | 说明 |
|------|------|
| 触发器 | 始终显示当前选中项 + 对应未读数红点 |
| "All Message" | 第一个选项，未读数为所有游戏合计；包含系统消息（🔔 图标区分，无独立筛选入口） |
| 游戏选项 | 各游戏图标 + 名称 + 各自未读数 |
| 默认选中 | 无游戏参数时默认 "All Message"；有游戏上下文时自动选中对应游戏 |
| 空状态 | 按选中的筛选值展示对应游戏的空状态和CTA |

**URL 参数：** `?filter={all|gameKey}&conv={convId}`
- 默认 `filter=all`，用户切换筛选时更新 URL 参数
- 从游戏页面进入聊天时，自动带入 `?filter={gameKey}`（参考导航栏动态切换逻辑中的游戏参数带入）
- 系统消息深度链接指向 `?filter=all`

## 一、页面概述

聊天页是交易双方沟通的核心页面，支持实时消息、交易状态同步和安全提醒。

---

## 二、页面结构

### 2.1 整体布局

**桌面端三栏布局：**
- 左侧：广告位
- 中间左侧：会话列表
- 中间右侧：聊天主区域
- 右侧：广告位

---

### 2.2 会话列表区

**标题栏：**
- "Message" + 未读消息总数徽章（如：10）

**系统消息项：**
- 🔔 图标 + 红点未读标识
- "System Message" 标签
- 时间戳
- 消息预览（如：📦 Vote for your favorite garden on PoliUnit an...）

**会话列表项：**

| 区域 | 内容 |
|------|------|
| 用户头像 | 圆形头像 |
| 用户信息 | 用户名 + 在线状态标签（Offline/Online） |
| 关联订单 | Trade #[订单编号] |
| 最后消息 | 消息预览（Trade Request / 文字消息） |
| 时间戳 | 日期时间（如：12/09 13:53） |

**列表排序：**
- 按时间倒序（最新消息在上）

---

### 2.3 聊天主区域

#### 头部信息栏

**用户信息：**
- 用户头像
- 用户名 + 在线状态标签（Offline/Online）
- 📅 可交易日期图标 + 日期列表（Mon, Tue, Wed, Thu, Fri, Sat, Sun）
- 🕐 可交易时间图标 + 时间段（Afternoon）

**操作按钮：**
- Report按钮（右上角）

#### 安全警告条（橙色背景）

- ⚠️ **Beware of scams!** 🙏
- 警告文字：Don't trust links or fake giveaways asking you to go off-site. Report anything suspicious!
- X 关闭按钮

#### 订单信息区

**标题行：**
- Trade #[订单编号]
- View Details 链接（右侧）

**道具展示：**
- 12格道具槽位（4x3网格）
- 显示交易相关道具图片

#### 消息区域

**交易请求消息（系统消息卡片）：**
- **Trade Request** 标签
- 提示文字：You have received a trade request. Do you accept? The trade will start immediately upon acceptance.
- **ACCEPT** 按钮（绿色）
- **DECLINE** 按钮（红色）

**普通文字消息：**
- 消息气泡
- 时间戳

#### 输入区域

- Message 输入框（placeholder: "Message"）
- 发送按钮（箭头图标，紫色背景）

---

### 2.4 Trade Chat Tips

**区域标题：**
- ⚠️ Trade Chat Tips

**提示列表（4条）：**
1. Before confirming, carefully double-check the item information and trade conditions.
2. Do not click any unknown third-party links. Confirm the user's Roblox ID matches their in-game ID.
3. Be polite and friendly to build good trading relationships.
4. If you have any doubts, communicate with your trade partner in the chat to confirm details.

## 三、功能说明

### 3.1 消息类型

**系统消息：**
- 交易请求（Trade Request）
- 订单状态变更通知
- 安全提醒

**用户消息：**
- 文字消息

### 3.2 交易请求处理

**收到交易请求时：**
- 显示Trade Request卡片
- 提示交易将立即开始
- **ACCEPT**：接受交易
- **DECLINE**：拒绝交易

### 3.3 举报功能

- 点击Report按钮
- 可举报可疑用户或消息

## 四、响应式设计

### 4.1 桌面端

- 三栏布局（广告 + 会话列表 + 聊天区 + 广告）
- 会话列表和聊天区并排显示

### 4.2 移动端

- 会话列表和聊天区分开为两个页面
- 点击会话进入聊天详情页
- 返回按钮回到会话列表
