# 交易链路优化 - 2026年3月

> 本文档是交易链路系统性优化的总纲，从核心问题诊断出发，串联所有优化需求，明确迭代路径与量化目标。
> 关联文档：信任体系与积分闭环-202603.md、聊天页-2025年11月更新.md（2026年3月迭代部分）

## 〇、为什么现在必须做这一轮优化

### 0.1 平台定位升级，交易体验必须跟上

- 我们已经从单一 GAG 交易站转型为**多游戏道具交易撮合平台**，新游戏陆续上线中
- 平台要在多游戏赛道站稳，核心竞争力就是**交易体验领先于竞品**——用户选择我们而不是其他平台，靠的是"更好找、更好谈、更放心"
- 当前交易链路中存在的体验短板（定价混乱、信任缺失、沟通效率低），如果不先解决，**每上线一个新游戏都会把同样的问题复制一遍**
- 现阶段用户规模适中，是打磨交易体验的最佳窗口——等用户量大了再改，成本和风险都会翻倍

### 0.2 内部数据信号

来自用户调研（N=150+）和交易数据分析的关键信号：

| 数据指标 | 现状 | 说明 |
|---------|-----|------|
| 撮合效率满意度 | **6.87 / 10** | 用户调研，远低于理想水平 |
| Specific 交易定价过高比例 | **54.8%** | 超过一半的指定交易要价不合理，直接拉低撮合率 |
| Open Offer 低价报价比例 | **31%** | 近 1/3 的报价远低于合理区间，浪费卖家审核精力 |
| 新用户首次交易门槛 | 调研排名 **#1 痛点** | 72.7% 用户反馈"寻找特定交易"困难，46.9% 反馈"发布交易"困难 |
| Counter-Trade 需求 | 调研排名 **前列** | 用户强烈需要编辑/调价能力 |
| 信任因素影响交易决策 | 调研 **关键因素** | 缺乏评价维度，信任只能靠"猜" |
| 移动端用户占比 | **>50%** | 近一个月数据，移动端体验质量直接影响一半以上用户 |

### 0.3 核心问题归因：5 个方向

把上面的数据和日常观察到的问题对照来看，归为 5 个方向：

```
A. 入口与引导 ──→ 新用户不知道怎么找交易、怎么发交易
B. 定价与公平性 ─→ 超半数交易定价不合理，劣币驱逐良币
C. 匹配效率 ───→ 找不到目标交易，视觉区分不清
D. 交易管理 ───→ 不能编辑、接受后不回复、聊天体验差
E. 信任与激励 ──→ 无评价体系，用户无动力确认完成交易

A→B→C 是"交易前"的效率问题
D 是"交易中"的管理问题
E 是"交易后"的闭环问题
三段合力，形成完整的交易体验闭环。
```

### 0.4 优化逻辑：交易体验飞轮

这些优化不是零散的"修补"，而是围绕一条清晰的底层逻辑：

> **让用户更快找到交易 → 更准定价 → 更高效完成交易 → 更愿意回来交易**

```
         ┌─────────────┐
         │  A. 入口引导   │   ← 让用户进得来、看得懂
         └──────┬──────┘
                ▼
    ┌───────────────────────┐
    │ B. 定价公平 + C. 匹配效率 │   ← 让交易找得到、价格合理
    └───────────┬───────────┘
                ▼
    ┌───────────────────────┐
    │ D. 交易管理 + 聊天体验    │   ← 让交易谈得成、改得了
    └───────────┬───────────┘
                ▼
    ┌───────────────────────┐
    │ E. 信任体系 + 积分闭环    │   ← 让用户愿确认、敢再来
    └───────────┬───────────┘
                ▼
         ┌──────────────┐
         │ 用户留存 & 复购  │   → 回流到顶部，带动更多交易
         └──────────────┘
```

每一层的优化都为下一层创造更好的条件，最终形成正循环。

### 0.5 量化目标

| 指标 | 当前基线 | 目标（上线后 30 天） | 衡量方式 |
|------|---------|-----------------|---------|
| 撮合效率满意度 | 6.87/10 | **≥ 7.5/10** | 用户调研复测 |
| Specific 交易过高定价率 | 54.8% | **≤ 35%** | 定价 ratio 分布监控 |
| Open Offer 低价报价率 | 31% | **≤ 20%** | 报价 ratio 分布监控 |
| 交易确认完成率（点击 COMPLETE 比例） | 待埋点 | **提升 ≥ 30%**（相对值） | COMPLETE 点击率 vs 进入聊天的订单数 |
| 新用户首笔交易转化率 | 待埋点 | **提升 ≥ 20%**（相对值） | 注册→首次发布/接受交易的转化漏斗 |
| 用户 7 日回访率 | 待埋点 | **提升 ≥ 15%**（相对值） | 活跃用户回访率 |
| 移动端聊天页可用面积 | ~250px | **≥ 450px** | 聊天可视区域高度 |

## 二、优先级排序（两版方案供讨论）

### 方案 A：按交易漏斗自上而下（先修入口，再补闭环）

逻辑：从用户旅程的第一步开始修，确保每一步优化都能被下一步承接。优先解决"进不来、看不懂"的问题，再解决"价格乱、找不到"，最后补全"交易后"闭环。

| 阶段 | 需求项 | 价值说明 |
|------|--------|---------|
| **Sprint 1** | **A-1** 交易入口强化 & 文案调整 | 纯前端改动，0 依赖，直接提升发布/查找转化 |
| | **A-2** 新手引导入口（how to trade 链接） | 顺手做 |
| | **B-2** 订单卡片 Have/Want 标识强化 | 纯前端样式，与 A-1 同属"视觉清晰度"主题，合并迭代 |
| **Sprint 2** | **B-1** 成交概率提示 + 调价引导 | 前端计算 + 弹窗，直击 54.8% 过高定价率 |
| | **C-1** Offer List 排序 | 前端排序，与 B-1 同属"定价公平"主题 |
| **Sprint 3** | **D-2** 聊天页交易状态卡片融入对话流 | 前后端配合，释放移动端空间，为信任体系卡片铺路 |
| | **D-3** 聊天不活跃提醒 | 后端定时任务，与 D-2 可合并开发 |
| **Sprint 4** | **E-1** 用户主页数据优化 | 前端展示层，信任体系前置 |
| | **E-2** 交易评价系统 | 前后端配合，核心信任闭环 |
| | **E-3** 积分体系 + Bump | 后端逻辑较重，与 E-2 构成完整激励闭环 |
| **Sprint 5** | **D-1** 编辑交易 / 调价 | 实现难度最高，但有前面定价提示作为铺垫 |
| | **A-3** 新手引导系统 V1 | 前端弹窗 + 引导逻辑 |
| | **E-4** 引导横幅 + 动效 | 锦上添花，收尾 |
| **后续** | **C-2** 道具订阅 & 通知 | 需要后端订阅表 + 通知队列，独立性强，可灵活安排 |

**方案 A 优势：** 前 2 个 Sprint 全是前端改动，上线快、见效快，能快速拿到数据验证方向。

逻辑：优先做 "实现成本低 × 用户价值高" 的需求，让团队在最短时间内看到数据变化，建立信心后再啃硬骨头。

## 四、成交概率提示 + 调价引导（B-1）

### 1. 功能概述

在用户执行 **Post Trade（发布交易）** 或 **Make Offer（发起报价）** 时，系统根据交易面板左右两侧的价值比，实时评估交易的合理性，并在必要时通过弹窗拦截，引导用户调整定价，以提升撮合成功率并保护用户避免亏损。

**问题域：** B. 定价与公平性
**解决问题：** 54.8% 的 Specific 交易定价过高 + 31% 的 Open Offer 低价报价
**预期效果：** 过高定价率降至 ≤35%，低价报价率降至 ≤20%

> ps: 该功能为平台基础交易链路中的新增逻辑，但需要通过功能开关来控制其应用在哪些游戏上，如 BF 这类已有强价差判断逻辑的游戏，则不使用该功能。

### 2. 术语定义

| 术语 | 说明 | 对应UI位置 |
|------|------|-----------|
| **Offering Value** | 用户愿意付出的道具总价值 | 交易面板 **左侧**（"OFFERING"区域） |
| **Looking for Value** | 用户期望获得的道具总价值 | 交易面板 **右侧**（"LOOKING FOR"区域） |

### 3. 计算公式

**ratio = Offering Value ÷ Looking for Value**

- **Post Trade 场景**：ratio = 发布者的 Offering Value ÷ 发布者的 Looking for Value
- **Make Offer 场景**：ratio = 报价者的 Offering Value ÷ 原帖发布者的 Offering Value（报价者出价对标的是原帖左侧道具）

### 4. 拦截阈值与行为

#### 4.1 阈值总览

| ratio 区间 | 状态 | 行为 | 拦截类型 |
|-----------|------|------|---------|
| 0.80 – 1.20 | ✅ Fair Zone（合理区间） | 无任何提示，正常提交 | 无 |
| 1.20 – 1.60 | ⚠️ Loss Warning（亏损预警） | 弹出温和提示弹窗 | 软拦截（可跳过） |
| > 1.60 | 🔴 Significant Loss（严重亏损） | 弹出强拦截弹窗 | 强拦截（需确认） |
| 0.50 – 0.80 | ⚠️ Low Match Probability（低匹配率） | 弹出温和提示弹窗 | 软拦截（可跳过） |
| < 0.50 | 🔴 Very Low Match（极低匹配率） | 弹出强拦截弹窗 | 强拦截（需确认） |

#### 4.2 设计说明

- **Fair Zone 上下各 20%**：避免对正常活跃用户造成高频打扰
- **亏损侧（ratio > 1.20）**：保护用户"给多了"，左侧价值显著高于右侧
- **低匹配侧（ratio < 0.80）**：提示用户"要多了"，右侧期望过高导致匹配概率低
- **不对称阈值设计**：亏损侧强拦截门槛（1.60）比低匹配侧（0.50）更紧，因为亏损是即时损失，而低匹配只是效率问题；同时低匹配侧阈值更宽松，允许用户适当抬价快速试探市场

### 5. 弹窗内容设计

#### 5.1 亏损预警弹窗（ratio > 1.20）

**标题**：⚠️ You might be giving away too much

**内容**：
- 提示文案：`Your Offering Value is significantly higher than what you're asking for. You may want to adjust your trade to avoid a potential loss.`
- 显示当前 ratio（如 `Your offer is worth ~1.4x what you're asking for`）
- 展示「已成交订单参考」区域（见 5.3）

**按钮**：
- `Adjust Trade`（主按钮，返回编辑）
- `Post Anyway`（次按钮，确认提交；强拦截时文案改为 `I understand, post anyway`）

#### 5.2 低匹配率提示弹窗（ratio < 0.80）

**标题**：📊 Low match probability

**内容**：
- 提示文案：`Based on recent market data, your asking price is higher than typical trades for these items. Consider adjusting to improve your chances of finding a match.`
- 显示当前 ratio（如 `Your offer covers ~60% of what you're asking for`）
- 展示「已成交订单参考」区域（见 5.3）

**按钮**：
- `Adjust Trade`（主按钮，返回编辑）
- `Post Anyway`（次按钮，确认提交；强拦截时文案改为 `I understand, post anyway`）

#### 5.3 已成交订单参考（Recent Trades Reference）

弹窗中包含一个参考区域，展示目标道具（Looking for 侧 / 对标道具）最近的成交记录，帮助用户理解市场行情：

- 展示最近 **3 笔已完成交易**
- 每条记录显示：交易双方道具概要 + 成交时间
- 若无足够成交数据，显示 `Not enough recent trade data for reference`
- 数据来源：系统内已完成（completed）状态的订单

### 6. 特殊场景处理

#### 6.1 Open Offer（无明确 Looking for）

当用户发布 Open Offer（不指定 Looking for 侧道具）时：
- **ratio 无法计算**（分母为 0 或不存在）
- **不触发任何拦截**，允许自由发布
- 后续可考虑基于道具历史成交均价给出参考提示（P2）

#### 6.2 Value 未知的道具

当交易涉及系统中尚无估值（value = null）的道具时：
- 若双侧均有未估值道具，**不触发拦截**
- 若仅单侧有未估值道具，按已知侧价值做有限判断，但**放宽阈值**（如 Fair Zone 扩大到 0.60 – 1.40），避免误拦截

### 7. 技术要点

- Value 数据来源：`items` 表的 `value` 字段
- 计算在前端完成（提交前拦截），不依赖后端接口
- 已成交订单参考数据通过查询 `completed` 状态订单获取
- 阈值参数建议做成可配置项，便于后续根据运营数据调整
- **功能开关**：通过配置控制是否对特定游戏启用（如 BF 已有强价差逻辑则不启用）

### 8. 后续迭代方向（P2）

- 根据上线后数据分析，优化阈值参数
- Open Offer 场景增加基于历史均价的参考提示
- 结合用户交易经验等级，对老用户适当放宽拦截
- 在交易面板实时显示 ratio 指示器（如颜色条），提交前即可感知

## 五、交易管理优化（D-1 / D-3）

> 聊天页对话流改造（D-2）详见「聊天页-2025年11月更新.md」中 2026年3月迭代部分。

### 5.1 编辑交易 / 调价功能（D-1）

**问题域：** D. 交易管理
**解决问题：** 用户发布后发现定价不合理或需要调整，目前只能取消重发，流程繁琐且丢失已有 Offer
**调研佐证：** Counter-Trade 需求在用户调研中排名前列
**预期效果：** 减少重复发布，提升用户体验

**功能设计：**

1. **可修改范围（V1）：**
   - ✅ 允许修改 **Want 侧**（Looking for）的道具和数量
   - ✅ 允许修改价值门槛
   - ❌ **Have 侧（Offering）不可修改**——防止"钓鱼"行为（先用好道具吸引报价，再换成差道具）

2. **已有 Offer 处理：**
   - 编辑后自动通知所有已报价用户："The trade you offered on has been updated. Please review the changes."
   - 已有 Offer 不自动作废，但报价方可选择撤回

3. **订单状态：** 编辑不改变订单状态（仍为 active）

**实现难度：** D（前后端配合，需要处理通知、状态一致性、防钓鱼逻辑）
**工作量估算：** 2-4 周

### 5.2 聊天不活跃提醒（D-3）

**问题域：** D. 交易管理
**解决问题：** 用户接受交易后不回复消息，对方长时间等待
**预期效果：** 提升交易中双方的响应速度

**功能设计：**
- 接受交易后，若一方在 **2 小时内未发送任何消息**，系统在聊天中发送提醒卡片
- 提醒文案：`⏰ Reminder: {username} is waiting for your reply. Keep the conversation going to complete this trade!`
- 超过 **12 小时无回复**，发送第二次提醒，附带取消提示
- 每笔交易最多发送 **2 次提醒**

**实现难度：** B（后端定时任务 + 系统消息）
**工作量估算：** 2-3 天

## 六、道具订阅 & 通知（C-2）

**问题域：** C. 匹配效率
**解决问题：** 用户想买某个道具但当前没有合适交易，无法被动等待通知
**预期效果：** 提升精准匹配效率，提高回访率

**功能设计：**

- 在单道具交易列表页增加 **"Subscribe"** 按钮
- 用户订阅后，该道具有新交易发布时自动发送通知
- **V1：** 站内通知（系统消息中心）
- **V2：** 可与 Discord 消息打通，用户授权后交易消息和订阅消息同步发送至 Discord

**实现难度：** C（需要后端新增订阅表和通知队列）
**工作量估算：** 1-2 周
