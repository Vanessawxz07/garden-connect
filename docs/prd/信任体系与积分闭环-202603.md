# 信任体系与积分闭环 PRD - 2026年3月

> 独立文档，与「交易链路优化-202603」并行。

---

## 一、需求背景与目标

### 1.1 背景

- 用户主页仅展示基础数据（完成交易数、粉丝数、宠物数），维度单一，缺乏主观评价维度。
- 用户在游戏内完成交易后，缺乏回到平台点击"确认完成"的动力，导致平台数据断层、挂单大厅信息陈旧。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| 完善客观展示 | 优化用户主页数据模块，让买家能快速评估卖家活跃度和靠谱程度 |
| 跑通评价闭环 | 建立基于"积分换曝光"的最简激励机制，提升交易确认率和真实评价数 |
| 沉淀核心资产 | 为未来多游戏交易、高级勋章等复杂系统打下扩展性基础 |

## 二、阶段一：用户信息展示优化（P0 - 前端展示层）

### 2.1 优化"客观数据"卡片栏

对用户主页（Profile）的数据卡片进行重构：

**合并冗余项：** 将现有的"Completed Trades"和"Successful Trades"合并，只保留一个清晰定义的 **"成功交易数（Successful Trades）"**。

**新增关键指标：**

| 指标 | 说明 | 无数据时展示 |
|------|------|------------|
| 好评率 (Positive Rate) | 好评数 / 总评价数 × 100% | `N/A` |
| 注册时长 | 强化为 `Member since {Year}` 格式 | — |

**卡片顺序（从左到右）：**

```
成功交易数 | 好评率 | 粉丝数 | 持有宠物数
```

### 2.2 新增"交易活跃度日历"组件

- **形式：** GitHub Contribution 风格的方格矩阵热力图
- **数据：** 统计该用户过去 **30 天**内，每天"发布挂单"和"完成交易"的总次数
- **展示：** 用颜色深浅直观展示近期活跃状态，无需复杂交互
- **位置：** 数据卡片下方
- **价值：** 防伪造成本极高，一眼识别"平时不活跃、突然挂极品道具"的潜在骗子账号
- **数据来源：** 使用现有系统日志数据

## 三、阶段二：交易评价系统（P1 - 核心逻辑层）

### 3.1 评价触发时机

基于现有交易确认流程，评价在 **订单转为 complete 状态后** 触发：

```
订单详情页 Accept → 进入聊天 → 卖家接受 → 双方出现 [START TRADE] / [TRADING GUIDE]
→ 任一方点击开始交易 → 出现 [COMPLETE] / [HAVING AN ISSUE]
→ 一方点击 COMPLETE（或超过 N 天未操作自动 complete）→ 订单转为 complete
→ 弹出评价弹窗
```

**注意：** 超时自动 complete 的订单，不弹出评价弹窗，但用户可在订单详情页手动补评。

### 3.2 评价内容设计

#### 3.2.1 评价等级

采用二元评价（参考截图及闲鱼风格）：

| 等级 | 图标 | 说明 |
|------|------|------|
| 👍 Positive | 笑脸 | 好评 |
| 👎 Negative | 哭脸 | 差评 |

> 初期不设"中评"，保持极简。好评和差评的行为规则差异化处理。

#### 3.2.2 好评标签（正面，可多选）

提供 **6 个预设标签**，用户可勾选 0~多个：

| 标签 | 英文 | 含义 |
|------|------|------|
| ⚡ Fast Trader | Fast Trader | 交易速度快，不拖延 |
| 😊 Friendly | Friendly | 沟通友好有礼 |
| 💰 Fair Deal | Fair Deal | 定价合理，不虚标 |
| 🤝 Trustworthy | Trustworthy | 值得信赖，说到做到 |
| 🎯 Smooth Trade | Smooth Trade | 全程顺畅无争议 |
| 💬 Responsive | Responsive | 回复及时，沟通效率高 |

#### 3.2.3 差评规则

- **不提供默认标签**
- **必填文本说明**：用户必须手动填写差评原因（文本框，最少 10 字符，最多 500 字符）
- **选填图片举证**：支持上传最多 **3 张** 截图作为附件（选填）
- **目的：** 提高差评门槛，减少无理由恶意差评，同时确保差评有据可查

#### 3.2.4 评价展示

- 评价在对方用户主页公开展示（按时间倒序）
- 评价人信息展示：显示用户名 + 头像（不支持匿名）
- 差评附带的文字和图片完整展示

### 3.3 评价修改与申诉

- **评价一旦提交不可修改**
- 被评价方可通过 **Report** 功能举报不实评价，由管理员介入处理
- 后续迭代（P2）考虑追评功能

## 四、阶段二：积分体系（P1 - 核心闭环）

### 4.1 积分单位

系统代币名称：**Points（Pts）**

### 4.2 积分获取机制

#### 4.2.1 新手破冰任务（一次性）

| 行为 | 积分 | 说明 |
|------|------|------|
| 首次登录/注册 | 20 Pts | 通过 Roblox OAuth 或常规注册，首次登录发放 |
| 首次发布挂单 | 10 Pts | 发布第一条有效交易请求 |

#### 4.2.2 日常活跃奖励

| 行为 | 积分 | 规则 |
|------|------|------|
| 每日签到 | **5 Pts** | 每天首次访问时点击签到按钮领取，固定 5 分，无连续加成 |

#### 4.2.3 交易闭环奖励（核心引擎）

| 行为 | 积分 | 说明 |
|------|------|------|
| 确认交易完成（点击 COMPLETE） | **20 Pts** | 任一方先确认即可获得 |
| 提交评价 | **20 Pts** | 确认后提交评价额外获得 |
| 双方确认尾款分 | **10 Pts** | 双方都点击 COMPLETE 后，两人各额外获得 |

**单笔交易最大积分收益 = 20（确认）+ 20（评价）+ 10（尾款）= 50 Pts**

### 4.3 积分防刷规则

| 规则 | 限制 |
|------|------|
| 每日签到 | 固定 5 Pts，无连续加成 |
| 每日积分上限 | **300 Pts**（含所有来源） |
| 同一对用户每日互评积分上限 | **100 Pts**（待定，防刷号对刷） |

### 4.4 积分消耗机制：Bump 功能

#### 4.4.1 功能定义

允许用户消耗积分，将自己的某条历史挂单重新"擦亮"，提升其在挂单大厅的排序位置。

| 参数 | 值 |
|------|-----|
| 消耗积分 | **50 Pts / 次** |
| 触发位置 | "我的挂单（My Offers）"列表，每条挂单卡片上的 [🚀 Bump] 按钮 |
| 冷却时间 | 同一条挂单 **每 24 小时** 最多 Bump **1 次** |

#### 4.4.2 排序逻辑

挂单大厅排序规则变更：

```
排序字段 = MAX(bumped_at, created_at) DESC
```

- **Bump 不是永久置顶**，而是等同于刷新了"更新时间"，排到最前面
- 随着时间推移和其他用户的新挂单/Bump，自然下沉
- 新发布的挂单天然排在前面（created_at 即为最新时间）

#### 4.4.3 前端交互

- **积分充足：** 按钮可点击，点击后确认弹窗 → 扣分 → 成功提示
- **积分不足：** 按钮置灰，Hover 提示 `Complete trades or check in daily to earn more Points`
- **冷却中：** 按钮置灰，显示剩余冷却时间 `Bump available in 12h 30m`

### 4.5 积分流水记录

**所有积分变动必须记录流水**，字段包括：

| 字段 | 说明 |
|------|------|
| user_id | 用户 ID |
| amount | 变动数量（正数为获取，负数为消耗） |
| balance_after | 变动后余额 |
| action_type | 来源动作枚举（check_in / trade_confirm / review / both_confirm_bonus / first_login / first_trade / bump） |
| reference_id | 关联对象 ID（如 trade_id） |
| created_at | 发生时间 |

## 五、关键触点引导（体验缝合）

### 5.1 全局积分展示

- **导航栏：** 登录用户在顶部 Navbar 显示当前积分余额，格式：`🪙 35 Pts`
- 点击跳转至积分明细页面

### 5.2 交易聊天框引导横幅

在待确认订单（已出现 COMPLETE 按钮）的聊天界面顶部，增加引导横幅：

```
💡 Confirm this trade and leave a review to earn 50 Points for free Bumps!
```

### 5.3 动效正反馈

以下场景触发简易 **Confetti 撒花动画**：

- 签到成功
- 完成评价赚到积分
- Bump 成功

## 六、数据模型概要

### 6.1 新增表

#### `point_balances` - 积分余额

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | PK |
| user_id | uuid | 用户 ID（唯一） |
| balance | integer | 当前余额，默认 0 |
| updated_at | timestamptz | 最后更新时间 |

#### `point_transactions` - 积分流水

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | PK |
| user_id | uuid | 用户 ID |
| amount | integer | 变动数量（正/负） |
| balance_after | integer | 变动后余额 |
| action_type | text | 来源动作枚举 |
| reference_id | uuid | 关联对象 ID（nullable） |
| created_at | timestamptz | 发生时间 |

#### `trade_reviews` - 交易评价

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | PK |
| trade_id | uuid | 关联交易 ID |
| reviewer_id | uuid | 评价人 |
| reviewee_id | uuid | 被评价人 |
| rating | text | `positive` / `negative` |
| tags | text[] | 正面标签数组（负面为空） |
| comment | text | 文字说明（负面必填，正面选填） |
| image_urls | text[] | 举证图片 URL 数组（最多 3 张） |
| created_at | timestamptz | 评价时间 |

#### `daily_check_ins` - 签到记录

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | PK |
| user_id | uuid | 用户 ID |
| check_in_date | date | 签到日期（user_id + date 唯一） |
| created_at | timestamptz | 签到时间 |

#### `trade_bumps` - 擦亮记录

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | PK |
| user_id | uuid | 用户 ID |
| trade_id | uuid | 挂单 ID |
| bumped_at | timestamptz | 擦亮时间 |

### 6.2 现有表变更

#### `trades`（或等效挂单表）

- 新增字段：`bumped_at` (timestamptz, nullable) — 最后擦亮时间
- 排序逻辑：`ORDER BY COALESCE(bumped_at, created_at) DESC`
